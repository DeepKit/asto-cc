## 一、ASTO 是什么

`ASTO` 是"属集变迁存在论"。

新版 ASTO-min 不是从上游 `DM` 推出下游结论的合法性转换器，而是一套跨场景结构编码语法。

最短句：

> **ASTO 负责把场景压成 `state / transition / actor / boundary_condition / exception` 五字段结构卡。**

再压一句：

> **ASTO-min 把场景压成五字段结构卡；DM-min 是可选的哲学来源，不是使用前置条件。**

---

## 二、ASTO 解决什么问题

`ASTO` 当前主要解决 6 类问题：

1. 提供统一的状态语言 — 让不同场景（工程、治理、AI、组织管理）使用同一套状态描述
2. 提供角色、边界与条件语言 — 明确谁在扰动、谁被扰动、边界在哪里
3. 提供变迁、例外与节律语言 — 描述属集如何重组、何时例外、如何循环
4. 把复杂案例压缩成可复述的结构编码 — 先使用 ASTO-min 五字段，必要时再附扩展坐标
5. 为下游治理与工程层提供结构上下文 — ARBT/TAT/ODD/COP/PFM/LSM 可读取，但必须各自终裁
6. 让不同场景使用同一套桥接语法对话 — 避免每个领域各说各话

## 三、ASTO 的最小硬核

ASTO-min 的核心不是 **1-5-6-7-1 螺旋**，而是五字段结构卡。1-5-6-7-1、五态、六阶、七序只作为旧 strong-ASTO 的扩展坐标系保留。

机器可读 v0.1 工件位于 `EE-学术发表层/ASTO-min-spec/`，包含 JSON Schema、YAML 模板、SHACL shape、四类案例和本地校验脚本；它只硬化 ASTO-min 五字段，不把 Core Profile 变成必填项。

### ASTO-min 五字段

| 字段 | 作用 |
|------|------|
| `ASTO.state` | 登记当前对象、角色、过程或系统的可识别状态 |
| `ASTO.transition` | 描述状态之间的变化、转移或路径 |
| `ASTO.actor` | 标明参与、触发、承接或影响变化的角色 |
| `ASTO.boundary_condition` | 标明状态、角色或过程成立的范围条件 |
| `ASTO.exception` | 标明使常规转移规则失效、暂停或升级的条件 |

### 旧扩展坐标：一元、五态、六阶、七序

以下内容只作扩展 profile，不是 ASTO-min 必填字段。

#### 一元：当前存在切片

任何时候，一个观察者面对的"当前状态"——一组属集的快照。这是螺旋的起点和每次回归的基准面。

#### 五态：空间性模态（可跳跃，需观察者确认）

五态源于 DM.P03「暂稳态与可转换性」的应用编码化——DM 问"结构为什么持存"，ASTO 答"当前该做什么"。详见 `DM.P03.本体.暂稳态与可转换性.md` §跨层映射。

属集在空间中可能处于的五种组织形态：

| 态 | 名称 | 特征 | 下游层的使用 |
|----|------|------|-------------|
| 1 | 自在 | 模糊感知，属集模式未明确 | PFM 寻域阶段的起点 |
| 2 | 共识 | 口头约定，属集模式被社会接受 | PFM 定人/港口阶段 |
| 3 | 编码 | 形式化，属集被写成规则 | COP 状态编码的输入 |
| 4 | 物化 | 物理/执行实现，规则被固化为工具 | ODD 封存的对应 |
| 5 | 定向 | 自我演化，包含"何时改规则"的元规则 | ARBT 元治理的起点 |

五态允许跳跃——一个创新可以直接从"自在"跳到"编码"，不必经过"共识"。

#### 六阶：时间性阶段（允许回退纠错）

属集在时间中变迁的节律：

混沌 → 秩序 → 流变 → 脉冲 → 崩解 → 归元

#### 七序：行动性操作（严格性较高，关键跳跃需下游审议）

具身 {感知, 解析, 干预, 设计, 物化, 回溯, 消解}

### 扩展编码纪律

- ASTO-min 正式案例至少标注 `state / transition / actor / boundary_condition / exception`
- 五态、六阶、七序只在需要更细坐标时附加
- 若当前阶段判断不稳，必须显式标记 `暂定 / 待复核`
- 若跨层跳跃会带来不可逆影响，ASTO 只标记 `exception`，由 TAT/ODD/COP 等下游决定处理状态

也就是说：

> **五态、六阶、七序是 ASTO 的扩展编码 profile，不属于 ASTO-min 五字段。**

## 五、ASTO 的核心创新方向

如果把 ASTO 的强项压短，我会概括成 5 点：

### 1. 用"属集变迁"替代"现成状态"作为描述起点

世界不是由一组现成状态构成，而是存在切片在扰动下不断重组的过程。ASTO 把这个过程写成可编码、可分析的语言。

### 2. 五态六阶七序的三维编码体系

这是 ASTO 最核心的工具贡献：

- **五态**定义了认知与组织形态的**空间性模态**
- **六阶**定义了变迁动力的**时间性阶段**
- **七序**定义了介入行动的**操作性步骤**

三者联合，可以作为许多场景的扩展坐标，但不能替代下游的责任、治理、工程或实践判断。

### 3. 把哲学约束翻译成可执行的运行时语法

ASTO 不停留在哲学讨论，而是把场景先压成五字段结构卡；以下内容只属于旧扩展 profile：
- 边界优先级（禁行红线 > 复数性 > 动变性 > 效率）
- FREEZE 条件与触发协议
- 人类裁决权的操作化声明

### 4. 为下游层提供统一结构上下文

ARBT、TAT、ODD、COP、PFM、LSM 都可以从 ASTO 获取同一套结构编码，避免各层各自发明术语。

### 5. 把复杂案例压缩成可复述的结构编码

通过 ASTO-min 五字段，复杂案例可以先被压缩成一个结构卡；五态+六阶+七序只作为可选附加坐标。

## 六、ASTO 不是什么

当前必须明确，`ASTO` 不是：

- 与 `DM` 平行竞争的第二本体论  
  它不发明新地基，只翻译上游的地基承诺
- 只服务 AI 对齐的专用理论  
  AI 是 ASTO 的高压力应用场景之一，但不是唯一场景
- 可以脱离人类裁决独立运行的自动决策手册  
  人类保留最终裁决权，这是价值公设
- 已经完成大规模经验验证的闭合体系  
  它当前更适合被读成桥接语法研究纲领

更准确地说：

> **ASTO 目前是一个以体系化写法呈现的应用桥接框架，其价值不在于替代下游，而在于让下游使用同一种结构语言说话。**

## 七、失败线与降级路径

> 详见 `CC-独立理论层/ASTO.P22.[失败线]-失败线汇总-Failure-Lines-Summary.md`

### 编码语法失效

| 编号 | 失效条件 | 后果 |
|------|---------|------|
| F1 | 两个独立编码者对同一变迁产出不一致的编码结果（kappa < 0.6） | 编码协议不可复述 |
| F2 | 五字段中 ≥ 2 个在真实编码实践中被持续性跳过 | 最小字段集需缩限 |
| F3 | 1-5-6-7-1 循环中步骤在 ≥ 3 个独立应用中相互冲突 | 步骤系统需统一 |

### 桥接语法失效

| 编号 | 失效条件 | 后果 |
|------|---------|------|
| B1 | ASTO 编码输出在 ≥ 2 个下游层中被报告为"不可直接使用" | "桥接"降级为"可选参考描述" |
| B2 | DM 核心概念与 ASTO 编码术语之间存在不可桥接的语义差距 | DM↔ASTO 接口需重开 |
| B3 | AI agent 在未受训练的情况下无法稳定产生符合 ASTO 编码协议的变迁描述 | "可被 AI 编码"主张降级 |

### 整体失败线

| 失败条件 | 后果 |
|---------|------|
| 属集变迁编码在 ≥ 2 个领域中无显著更精确 | 不可替代性被削弱 |
| 涌现属性的正面建模无法区分"真涌现"和"预先存在但未记录的属性" | 涌现编码退化为分类误差 |
| 属集操作中 ≥ 2 个在实证中无法被独立编码者稳定区分 | 操作类型需合并 |
| 免疫宣言"自我否定条款"在 ≥ 2 次外部批评中被绕过 | 免疫宣言从边界声明降级为免责声明 |

### 最低可用口径

- **ASTO-min** = state / transition / actor / boundary_condition / exception
- 失效边界：ASTO 不输出形式验证结论、治理决策、因果归因或责任终裁

## 八、不可替代问题与外部比较

> 详见 `CC-独立理论层/ASTO.P21.[外部比较]-不可替代问题与外部比较-Irreplaceable-Problem-and-External-Comparison.md`

### ASTO 解决的核心问题

> **变化发生时，如何系统性地描述"什么东西变了"——不只是说"状态 A 变成状态 B"，而是跟踪属集内部哪些成员增删、重组、深化、湮灭，以及这种变化是否可复述、可比较、可转交给下一个解释者。**

### 为什么传统方法不够

| 问题 | 传统方法 | ASTO |
|------|----------|------|
| 系统从 v1 到 v2 "具体变了什么属性" | 状态转移图——只有节点名称变化 | 属集变迁序列——列出增/删/改/重组/深化/湮灭 |
| 两个架构师对同一变更的编码是否一致 | 没有标准 | 属集变迁语法提供可复述的编码协议 |
| 变化不只是状态切换，而是结构重组 | 状态机/工作流/时序逻辑只建模"切换" | 1-5-6-7-1 循环描述属集变迁 |
| 系统涌现新属性 | 需要预先定义所有可能状态 | 属集 open/closed + emergence 编码 |
| 需要把变化描述转交给下游 | 各层自建描述，语义断裂 | 统一变迁描述接口 |

### 不可替代性声明

> 如果去掉 ASTO，会失去：属集级别的变化粒度、可复述的编码协议、涌现属性的正面建模、向治理层的统一变迁描述接口。ASTO 不竞争"如何建模流程"，它解决的是"如何描述结构本身在变化中的重组"。

### 核心比较矩阵

| 维度 | ASTO | 状态机 | BPMN | 时序逻辑 | 事件演算 | 范畴论 | 系统动力学 |
|------|------|--------|------|---------|---------|--------|-----------|
| 核心关注 | 属集结构重组 | 状态转移规则 | 活动序列 | 命题时间约束 | 事件改变 fluents | 结构保持映射 | 存量-流量动力学 |
| 变化粒度 | 属集成员 | 状态节点 | 活动节点 | 命题真值 | fluent 真值 | 函子映射 | 变量值 |
| 结���变化建模 | 核心 | 否 | 否 | 否 | 否 | 是（抽象） | 否 |
| 涌现处理 | open/closed + emergence | 需预定义 | 需预定义 | 需预定义 | 需预定义 | 自然变换 | 不处理 |

### 外部关系定位

- 和所有现有建模方法论的共同点：都处理"变化"，但传统方法只描述"切换"而非"重组"
- ASTO **不是**替代品，而是补充层——在其他人描述"什么东西在流动"时，ASTO 描述"什么东西在结构上被重组"
- 可以从范畴论借用法则，但不声称形式等价——工具理性，不是数学乐观主义

### 最小独立闭环

ASTO 可独立成立；DM 只提供可选哲学解释。只要"变化需要被结构化描述"的需求存在，ASTO 就有独立问题域。

## 九、ASTO 与上下游各层的关系

当前公开主链是：

`DM -> ASTO -> ARBT -> TAT -> ODD`

其中：

- `DM` 提供哲学地基
- `ASTO` 提供桥接语法
- `ARBT` 提供文明与治理判断
- `TAT` 提供责任阈值
- `ODD` 提供工程闭环

它向下的转接作用限于提供结构上下文：

- 对 `ARBT`：推进到文明演化与权责边界（五态进入治理判断）
- 对 `TAT`：提供责任阈值与责任决定条件语法
- 对 `ODD / COP / PFM / LSM`：提供统一结构编码与边界前置字段

## 十、ASTO 的 DM 风险防火墙

### 最小依赖

即使暂不接受 `DM` 的强版本，ASTO 仍可先被读成：

- 一套跨场景结构语言
- 一套状态压缩与案例编码框架
- 一套用于分析、审议、设计与裁决的桥接语法

### ASTO 受争议时如何阅读

若 `DM` 的强形而上学承诺受争议，`ASTO` 当前最稳的降级阅读方式是：

- 先把它读成一套实用的结构编码工具
- 先把它读成跨场景对话的统一语法
- 不把它强行读成必须依赖 DM 才成立的闭合理论

### 防火墙句

> **即使 `DM` 的强版本受压，`ASTO` 仍可作为跨场景结构编码工具成立；真正需要降级的，是它的最强哲学解释叙事，而不是它的全部编码与翻译能力。**

## 十一、ASTO 当前该怎么读

如果你是第一次接触 `ASTO`，更稳的阅读方式不是一下子吞完整套正文，而是分三步：

### 第一步：先读总览

先读本文件，确认：

- 它在解决什么问题
- 它不在解决什么问题
- 它最小硬核（五态六阶七序）是什么
- 它和上下游怎么衔接
